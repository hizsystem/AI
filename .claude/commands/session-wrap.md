# 세션 래핑 & 자기개선 루프

하루 작업을 마무리하며, **오늘 세션에서 배운 것을 시스템에 영구 반영**합니다.
Felix(Nat Eliason)의 "매일 밤 자기개선 루프"를 MarketingOS 버전으로 구현한 스킬입니다.

> 핵심 원칙: "같은 개입을 두 번 하지 않는다."
> 오늘 우성민님이 직접 손 댄 지점 → 내일부터는 시스템이 자동 처리

사용자 입력: $ARGUMENTS

## 0단계: 세션 데이터 수집

> **중요: 멀티 워크트리 환경**
> 우성민님은 클라이언트별 워크트리에서 여러 세션을 동시에 운영합니다.
> 이 스킬은 **어떤 워크트리에서 실행하든 전체 워크트리를 스캔**합니다.

### 자동 수집 (도구 사용)

1. **전체 워크트리 활동 스캔** — 모든 워크트리의 오늘 커밋을 한 번에 수집:
   ```bash
   # 메인 레포 (모든 브랜치 커밋)
   git -C /Users/wooseongmin/AI log --since="today 00:00" --oneline --all

   # 각 워크트리별 오늘 커밋 + 스테이지되지 않은 변경사항
   for wt in /Users/wooseongmin/AI/.claude/worktrees/*/; do
     name=$(basename "$wt")
     log=$(git -C "$wt" log --since="today 00:00" --oneline 2>/dev/null)
     diff=$(git -C "$wt" diff --name-only 2>/dev/null)
     if [ -n "$log" ] || [ -n "$diff" ]; then
       echo "=== $name ==="
       [ -n "$log" ] && echo "$log"
       [ -n "$diff" ] && echo "[uncommitted] $diff"
       echo
     fi
   done
   ```

2. **현재 메모리 인덱스** 읽기:
   - `/Users/wooseongmin/.claude/projects/-Users-wooseongmin-AI/memory/MEMORY.md` 읽기

3. **활성 클라이언트 컨텍스트** 확인:
   - 커밋/변경이 있는 워크트리에서 관련 `clients/{name}/context/` 폴더 식별

### 사용자 입력 처리

없으면 아래 질문을 합니다 (최대 3개, 해당하는 것만):

> **오늘 전체 세션을 래핑합니다. 간단히 알려주세요:**
> 1. 오늘 여러 세션에서 **직접 수정하거나 다시 지시한 것**이 있나요? (어느 워크트리든)
> 2. 오늘 **새로 알게 된 것** 중 기록할 만한 것이 있나요? (클라이언트, 프로세스, 도구 관련)
> 3. 오늘 **반복 작업** 중 자동화하고 싶은 것이 있나요?

사용자가 "없어" / "패스" / 빈 입력이면 → git 데이터만으로 분석을 진행합니다.

## 1단계: 개입 지점 분석 (핵심)

에이전트를 호출하여 오늘 세션을 분석합니다.

### Agent: 🔄 세션 분석가 (session-analyst)

```
당신은 MarketingOS의 세션 분석가입니다.
오늘 하루 **전체 워크트리**의 작업 데이터를 분석하여, "운영자(우성민님)가 직접 개입해야 했던 지점"을 추출합니다.

## 오늘의 데이터:

{각 워크트리의 git log + uncommitted changes}

## 분석 프레임워크:

### A. 개입 지점 추출
아래 5가지 유형으로 분류하여 찾습니다:

1. **반복 수정 (Repeated Correction)**
   - 같은 종류의 지시를 2회 이상 한 것
   - 에이전트 산출물을 받고 다시 수정 요청한 것
   - 신호: 같은 파일이 짧은 간격으로 여러 번 수정됨

2. **맥락 누락 (Missing Context)**  
   - 클라이언트 정보를 매번 새로 알려줘야 했던 것
   - 이전 대화에서 이미 정한 걸 다시 설명한 것
   - 신호: context/ 폴더에 없는 정보를 대화에서 직접 제공

3. **품질 미달 (Quality Gap)**
   - 에이전트 결과물의 톤/깊이/정확도가 기대에 못 미친 것
   - 직접 내용을 크게 수정한 것
   - 신호: 산출물 파일이 생성 직후 대폭 수정됨

4. **프로세스 병목 (Process Bottleneck)**
   - 수동으로 해야 해서 시간이 걸린 반복 작업
   - 자동화할 수 있었는데 안 된 것
   - 신호: 비슷한 패턴의 커밋이 여러 번

5. **판단 위임 불가 (Judgment Required)**
   - 에이전트에게 맡길 수 없어서 직접 판단한 것
   - 이건 "자동화 대상"이 아니라 "구조적으로 사람 판단이 필요한 것"
   - 신호: 전략적 의사결정, 클라이언트 감정 대응, 가격/범위 결정

### B. 각 개입 지점별 처리 방향 제안

각 개입 지점에 대해 아래 4가지 중 하나로 분류:

| 처리 유형 | 설명 | 자동 반영 |
|-----------|------|----------|
| **feedback 메모리** | 에이전트 행동 규칙 추가/수정 | O (승인 후) |
| **context 업데이트** | 클라이언트 컨텍스트 보강 | O (승인 후) |
| **스킬/에이전트 수정** | 스킬 프롬프트나 에이전트 정의 수정 | 제안만 |
| **크론/자동화 전환** | 반복 작업을 스크립트로 분리 | 제안만 |

### C. 클라이언트 인터랙션 로그 (워크트리 단위)

오늘 활동이 있는 **각 워크트리**별로:
- **워크트리명**: 어떤 클라이언트/프로젝트 작업인지
- **커밋 요약**: 무엇을 했는지 (1줄)
- **새로 알게 된 정보**: 있으면
- **context/ 업데이트 필요**: 있으면
- **크로스 워크트리 연관**: 다른 워크트리 작업에 영향을 주는 것 (예: main에서 스킬 수정 → huenic 워크트리에 영향)

### D. 역할 분리 체크

오늘 작업 중 아래에 해당하는 것이 있는지 확인:
- "이건 AI 판단이 필요 없는 반복 실행이다" → 크론/스크립트 후보
- "이건 현재 에이전트 A가 하는데 에이전트 B가 더 적합하다" → 라우팅 수정 후보
- "이건 하나의 스킬이 너무 많은 걸 한다" → 분리 후보
```

## 3단계: 실행

사용자 승인을 받은 항목에 대해 **실제로 반영**합니다.

### feedback 메모리 생성/업데이트

기존 메모리 파일과 중복 확인 후:

- **신규**: `/Users/wooseongmin/.claude/projects/-Users-wooseongmin-AI/memory/feedback_{slug}.md` 생성
- **업데이트**: 기존 파일 수정
- **MEMORY.md**: 인덱스에 추가/업데이트

메모리 파일 형식 (기존 패턴 준수):
```markdown
---
name: {메모리 이름}
description: {1줄 설명 — 미래 대화에서 관련성 판단용}
type: feedback
---
{규칙/사실}

**Why:** {왜 이 규칙이 필요한지 — 오늘 있었던 일 기반}

**How to apply:** {언제/어디서 이 규칙을 적용하는지}
```

### context 업데이트

- 해당 클라이언트의 `clients/{name}/context/` 파일 수정
- context 폴더가 없으면 생성 제안 (자동 생성하지 않음)
- 변경 내용을 명확히 diff로 보여준 후 반영

### 인터랙션 로그 기록

각 클라이언트 폴더에 `interaction-log.md` 파일이 있으면 추가, 없으면 생성 제안:

```markdown
## {YYYY-MM-DD}
- **작업**: {1줄 요약}
- **새 정보**: {있으면}
- **다음 액션**: {있으면}
```

## 주의사항

1. **"I will do X" 금지** — 메모리 파일을 "생성하겠습니다"가 아니라 실제 생성 후 "생성했습니다"로 보고
2. **중복 메모리 금지** — 반드시 기존 MEMORY.md와 메모리 파일들을 확인한 후 신규/업데이트 판단
3. **과잉 기록 금지** — 코드에서 직접 읽을 수 있는 정보(파일 구조, git 히스토리)는 메모리에 넣지 않음
4. **클라이언트 context 직접 수정 시 반드시 diff 표시** — 승인 없이 context 파일 수정 금지
5. **개입 지점이 0개여도 OK** — "오늘은 시스템이 잘 돌아갔습니다"도 유의미한 결과
6. **판단 위임 불가 항목은 자동화 대상이 아님** — 기록은 하되 "사람 판단 필요" 태그를 명확히
