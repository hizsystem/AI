---

## 1. 왜 하는가

### 부회장님 방침 요약
- 조직이 빠르게 변하는 시기, **비효율 개선 + 자기주도적 성장**에 투자
- 개발자들의 오픈소스처럼 **회고와 레슨런을 공유하여 상향 평준화**
- **BC3팀이 선도하여** 전사에 전파

### 기대 효과
| 구분 | Before | After |
|------|--------|-------|
| 보고 | 위→아래 일방향 | 수평 공유, 누구나 배움 |
| 실패 | 숨기거나 반복 | 기록 → 전파 → 재발 방지 |
| 의사결정 | 결과만 남음 | **왜 그 결정을 했는지** 맥락 보존 |
| 성장 | 개인에 의존 | 팀 전체가 함께 레벨업 |

---

## 2. 3가지 체계

### 체계 ① — 금요 위클리 (Weekly Share)

**목적**: 주간 보고를 '공유'로 대체
**빈도**: 매주 금요일 (비동기 OK)
**소요시간**: 작성 10분 / 읽기 5분

#### 포맷 (Slack 또는 Notion)

```
📅 BC3 Weekly Share — {날짜}
작성자: {이름}

🎯 이번 주 핵심 성과 (1~3개)
-

📚 배운 것 / 깨달은 것
-

⚠️ 막혔던 것 / 실패한 것
-

🔜 다음 주 포커스
-

🤝 도움 요청 / 공유 사항
-
```

#### 운영 규칙
- 분량: **5줄 이내** 권장 (길면 안 읽음)
- 월요일 아침에 읽고 이모지 리액션으로 확인 표시
- 리더가 먼저 작성 (문화 선도)

---

### 체계 ② — 프로젝트 회고 (KPT + Before/After)

**목적**: 프로젝트/마일스톤 완료 후 학습 정리
**빈도**: 프로젝트 종료 시 or 월 1회 (진행 중인 경우)
**소요시간**: 30~45분

#### 템플릿

```

📅 회고일: {날짜}
```

### 체계 ③ — 오픈소스 공유 (Lessons Learned)

**목적**: 팀의 학습을 전사에 전파하여 상향 평준화
**빈도**: 수시 (위클리에서 발굴 → 정제 → 전사 공유)
**채널**: #company-learnings (전사) + #bc3-learnings (팀)

#### 공유 포맷 (Slack 포스팅용)

```
🔖 [BC3] {제목}
📅 {날짜}
🎯 한 줄 요약: {한 문장}

📝 상세:
  - Before: {이전 상태/방법}
  - After: {개선된 상태/방법}
  - 배운 점: {핵심 인사이트}

💡 다른 팀에 추천: {바로 적용 가능한 팁}
```

#### 공유 기준
- "이거 다른 팀도 알면 좋겠다" 싶으면 공유
- 실패 사례도 환영 (오히려 더 가치 있음)
- 짧을수록 좋음 — 읽는 데 2분 이내

## 6. 핵심 원칙

1. **짧고 자주** > 길고 가끔 — 15분 위클리가 2시간 월간보다 효과적
2. **실패에 박수** — "공유해줘서 고마워"가 기본 반응
3. **리더가 먼저** — 리더가 본인의 실패를 먼저 공유해야 문화가 됨
4. **비동기 우선** — 바쁘면 슬랙, 주말도 OK (부회장님 방침)
5. **의사결정의 '왜'를 기록** — 코드는 "무엇"을, 회고는 **"왜"**를 남긴다
